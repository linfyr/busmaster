# BUSMASTER 新增硬件适配说明

## 1. 项目现状概览（面向硬件扩展）

BUSMASTER 当前是典型的 **上层应用 + 驱动抽象层（DIL）+ 硬件控制器实现（DLL）** 架构：

- 上层应用：`Sources/BUSMASTER/Application`（MFC 主程序，链接 DIL_Interface）
- 协议/驱动抽象层：`Sources/Kernel/BusmasterDriverInterface`
  - CAN 路由：`DIL_CAN.cpp/.h`
  - LIN 路由：`DIL_LIN.cpp/.h`
  - 统一接口导出：`Include/BusmasterDriverInterface.h`（`GetDilInterface`）
- 硬件实现层：`Sources/BUSMASTER/CAN_*`、`Sources/BUSMASTER/LIN_*`
  - 每个硬件模块最终以 DLL 形式被路由层动态加载（`LoadLibrary + GetProcAddress`）

## 2. 现有硬件适配机制

### 2.1 动态加载机制

以 CAN 为例，`DIL_CAN.cpp` 中通过：

1. 枚举/列表维护驱动项（`sg_ListDIL`）
2. `DILC_SelectDriver` 按 DriverID 选择 DLL 名称（如 `CAN_STUB.dll`）
3. 从 DLL 取导出符号：`GetIDIL_CAN_Controller`
4. 获取 `CBaseDIL_CAN_Controller` 实例并调用统一接口（初始化、列举硬件、收发、状态等）

LIN 机制同理，对应导出函数为 `GetIDIL_LIN_Controller`。

### 2.2 控制器实现接口约束

新增 CAN 硬件实现需要对齐 `CBaseDIL_CAN_Controller` 接口（文件：
`Sources/Kernel/BusmasterDriverInterface/Include/BaseDIL_CAN_Controller.h`），核心包括：

- 初始化/反初始化：`CAN_PerformInitOperations` / `CAN_PerformClosureOperations`
- 硬件枚举/选择：`CAN_ListHwInterfaces` / `CAN_SelectHwInterface`
- 启停与收发：`CAN_StartHardware` / `CAN_StopHardware` / `CAN_SendMsg`
- 客户端缓冲与注册：`CAN_ManageMsgBuf` / `CAN_RegisterClient`
- 运行状态与参数：`CAN_GetCntrlStatus` / `CAN_GetControllerParams` / `CAN_GetErrorCount`

## 3. 新增硬件适配实施路径（建议）

## 3.1 推荐路径 A：有硬件 SDK 且可编译源码

1. 以现有完整模块为模板（推荐从 `CAN_STUB` 或同类源码模块复制骨架）创建新目录：
   - 示例：`Sources/BUSMASTER/CAN_<VENDOR>_<MODEL>/`
2. 实现控制器类（继承 `CBaseDIL_CAN_Controller`），封装厂商 SDK。
3. 导出标准入口函数：
   - CAN：`GetIDIL_CAN_Controller(void** ppvInterface)`
   - LIN：`GetIDIL_LIN_Controller(void** ppvInterface)`
4. 在路由层注册新驱动：
   - `DIL_CAN.cpp` / `DIL_LIN.cpp` 中增加 DriverID、显示名称、`LoadLibrary` 分支。
5. 在构建系统接入子模块：
   - `Sources/BUSMASTER/CMakeLists.txt` 增加 `add_subdirectory(...)`
   - 新模块提供自己的 `CMakeLists.txt`（或保持与现有 vcxproj 方案一致）
6. 在 UI 侧验证可见性与可选性（驱动列表来自 DIL 列表）。

## 3.2 路径 B：仅有厂商已编译 DLL（无源码）

仓库中已有“仅拷贝 DLL 的模块”模式（例如 `CAN_PEAK_USB/CMakeLists.txt` 主要做复制）。
可沿用该模式：

1. 提供统一导出接口兼容层（若厂商 DLL 不能直接满足，需新增薄封装 DLL）。
2. 在构建/发布阶段将厂商 DLL 与封装 DLL 一起复制到 `BIN/<Config>/`。
3. 路由层仍只感知统一入口函数与抽象接口。

## 4. 新增 CAN 硬件的最小改动清单

至少涉及以下位置：

1. `Sources/Kernel/BusmasterDriverInterface/DIL_CAN.cpp`
   - 增加 DriverID
   - 增加显示名称（`sg_ListDIL`）
   - 增加 `LoadLibrary("<YourDll>.dll")` 分支
2. `Sources/BUSMASTER/CMakeLists.txt`
   - 增加新模块 `add_subdirectory`（若参与构建）
3. `Sources/BUSMASTER/CAN_<VENDOR>_<MODEL>/...`
   - 控制器实现 + Extern 头 + 导出函数
4. 需要时补充安装/分发脚本（`Installation Setup` 下相关脚本）

## 5. 适配时的注意事项

- **ABI/导出名必须稳定**：路由层按固定符号名 `GetIDIL_*_Controller` 加载。
- **错误回滚逻辑要保留**：`DIL_CAN.cpp` 已实现“新驱动加载失败回退旧驱动”机制，新增分支需保持一致行为。
- **通道上限与配置一致性**：注意 `SCONTROLLER_DETAILS`、通道数量、波特率/过滤器默认值与 UI 配置一致。
- **厂商驱动依赖**：优先在模块内部显式处理驱动库加载失败并返回可读错误。
- **Windows 依赖**：当前工程大量依赖 MFC/ATL、WinDDK 与厂商驱动，Linux 环境仅可做静态检查，完整验证需在 Windows 开发环境执行。

## 6. 建议的验证清单（新增硬件后）

1. 驱动列表可见且名称正确（DIL 列表/UI）。
2. 可枚举并选择目标硬件通道。
3. 启动/停止硬件成功，无资源泄漏。
4. 收发报文、时间戳、过滤器行为正确。
5. 与日志、回放、节点仿真等现有功能组合回归通过。
6. 回归 Simulation（`CAN_STUB`）与已有主流硬件，避免引入兼容性回退。

## 7. 本次分析结论

BUSMASTER 的硬件扩展点较清晰：**在 DIL 路由层注册 + 提供符合基类契约的硬件控制器 DLL** 即可接入。
后续新增硬件建议先按 CAN 路径落地一个最小可用版本（枚举/连接/收发），再逐步补齐高级能力（错误计数、过滤策略、多通道参数化）。
